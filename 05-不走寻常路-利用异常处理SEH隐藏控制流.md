# 不走寻常路：利用异常处理（SEH）隐藏控制流

我们平时看程序执行流程，脑子里通常都是一条非常“规矩”的路线：

```text
CALL -> 进入函数
JMP  -> 跳去别处
RET  -> 回来
```

这套理解在大多数普通程序里没问题。但在逆向对抗场景里，总有些家伙不爱走正门。它们会故意制造一次异常，让系统帮自己“拐个弯”把控制流送到另一个地方去。

如果你第一次碰到这种样本，很容易一脸问号：明明刚才执行了一条非法指令，怎么程序没死，反而继续跑出新逻辑了？这时你遇到的，多半就是基于 SEH 的控制流隐藏。它最抽象的地方就在这儿：看着像崩了，其实是在拐弯。

这一篇我们就从最基础的问题讲起：SEH 到底是什么，程序怎么注册异常处理函数，为什么它能被拿来藏控制流，以及在调试器里该怎么把这条“暗道”揪出来。

---

## 一、SEH 是什么？

SEH，全称 Structured Exception Handling，结构化异常处理。

简单理解，它是 Windows 提供的一套“程序出事后怎么办”的机制。

比如程序运行时可能遇到除零、非法内存访问、非法指令、断点异常这些情况。正常人看到“异常”两个字，第一反应通常是“程序崩了”。但对操作系统来说，异常并不等于立刻结束进程，而是会先尝试问一句：

**“有没有人注册过处理这个异常的函数？”**

如果有，系统会把控制权转交过去，让那个处理函数决定接下来怎么办。

这就是 SEH 的价值所在：它本来是为了提升程序健壮性，却也可以被逆向对抗者拿来当隐藏跳板。

---

## 二、最简单的生活化理解

你可以把 SEH 理解成公司里的应急联系人制度。

平时员工按正常流程办事，谁也不找应急联系人。但一旦出事，比如停电、跳闸、设备炸了，公司不会立刻宣布“今天全完了”，而是先看：

```text
这件事归谁处理？
```

如果早就登记了联系人，那系统就会把问题转过去。

程序里的 SEH 也是一样。正常情况下它按 `CALL`、`JMP` 这些显式路径老老实实往下跑，一旦“出事”，控制权就会交给异常处理函数。攻击者聪明的地方就在这儿：他知道分析者大多盯着正常路径看，于是故意把真正逻辑塞到“出事之后”的那条支线里。

---

## 三、SEH 在 32 位下是怎么挂上的？

在经典 32 位 Windows 里，SEH 链和线程环境有关，常见会通过 `FS:[0]` 访问。

你在汇编里经常能看到类似结构：

```asm
push handler
push dword ptr fs:[0]
mov  dword ptr fs:[0], esp
```

这三步放在一起看就很清楚了：`push handler` 是把异常处理函数地址压栈，`push fs:[0]` 是把原来的链表头先保存下来，`mov fs:[0], esp` 则让当前栈顶变成新的 SEH 记录头，也就是把新的异常处理器注册上去。所以你一旦看到 `FS:[0]` 这类操作，就该敏感起来，它十有八九和 SEH 注册有关。

---

## 四、最简单的 SEH 控制流例子

看一个很小的思路：

```asm
push handler
push fs:[0]
mov  fs:[0], esp

xor eax, eax
div eax

next:
...

handler:
; 在这里修改 EIP 或直接跳转到真正逻辑
...
```

这里最关键的是 `div eax`。

因为前面 `xor eax, eax` 之后，`eax = 0`，再执行 `div eax` 就会触发除零异常。

如果没有 SEH，这程序大概率直接崩。

但因为前面已经注册了异常处理器，所以系统不会立刻结束它，而是把控制权交给 `handler`。

于是原本表面上看像“程序出错了”，实际上却是：

**作者主动借异常完成了一次隐藏跳转。**

---

## 五、为什么这种手法会迷惑逆向分析？

因为它绕开了你最熟悉的显式控制流。

你在静态分析里最习惯看的，一般就是 `call xxx`、`jmp xxx`、`jz/jnz xxx` 这种显式跳转；而 SEH 这条线看起来却可能是这样的：

```text
正常路径上没有明显跳转
    ↓
执行一条会异常的指令
    ↓
系统介入
    ↓
转到异常处理器
```

也就是说，真正的控制流转移不是某条普通跳转指令直接告诉你的，而是“借操作系统的异常分发机制偷偷完成的”。所以你如果只盯 `jmp` 和 `call`，很容易被它带偏。

如果你只盯着显式分支，很容易以为这里已经“走死了”。

---

## 六、逆向里常见的 SEH 混淆手法

SEH 不只是拿来处理除零，作者通常会故意挑一些容易制造异常的操作。最直观的是除零，像：

```asm
xor ecx, ecx
div ecx
```

也可以是非法内存访问，例如：

```asm
mov eax, [0]
```

或者直接访问一个故意无效的地址。再狠一点，也可以故意塞非法指令，或者用某些 `int` 指令、断点指令之类的方式人工触发异常。作者这么干的好处其实很直接：静态控制流图会被搅乱，单步调试时你也容易被异常节奏打断，新手更是特别容易当场误判成“程序崩了”。

---

## 七、在汇编里怎么识别 SEH 注册？

最直接的信号就是和 `FS:[0]` 有关的操作。

例如：

```asm
push offset handler
push fs:[0]
mov  fs:[0], esp
```

后面退出时还可能有恢复动作，比如：

```asm
mov eax, [esp]
mov fs:[0], eax
add esp, 8
```

这说明程序在进入某段逻辑前先挂了一个新的异常处理器，用完之后又把原来的链恢复回去。看到这一组操作，基本就该意识到：后面大概率会故意出异常。

---

## 八、调试时为什么容易被 SEH 绕晕？

因为调试器也会参与异常处理流程。

程序触发异常后，一般不是简单地“继续执行下一条”，而是先被系统和调试器一起感知，然后调试器决定是否拦住；只有它放行了，程序自己的异常处理器才有机会接手。于是就会出现一个很常见的场面：你在调试器里一看，异常弹窗出来了，下意识觉得程序炸了；但如果你放过去，它反而会正常跑到隐藏逻辑。

也就是说，SEH 样本经常不是“真的崩”，而是“故意装作崩了”。别被那个异常弹窗当场劝退，这玩意很多时候只是烟雾弹。

---

## 九、那调试时该怎么抓它？

核心思路很简单：

**别只盯异常本身，要盯异常之后程序去了哪里。**

常见做法大体就是两条线：一条线盯注册点，一条线盯异常之后到底跑去了哪。比如如果你静态已经看到：

```asm
push offset handler
push fs:[0]
mov  fs:[0], esp
```

那这个 `handler` 地址就是关键目标，你完全可以直接在这里下断点。这样程序一旦通过 SEH 转进来，你就能第一时间截住。另一条线就是在调试器里合理处理异常：对于一些预期内异常，不要一看见就结束分析，而是根据设置去决定忽略一次、让程序自己的处理器先接，或者在特定异常类型发生时断下观察。不同调试器的选项名字可能不一样，但思想其实就一句话：**别让调试器把作者设计好的“异常跳转”提前掐死。**

---

## 十、一个最小实战分析思路

假设你看到这样一段代码：

```asm
push offset seh_handler
push fs:[0]
mov  fs:[0], esp

mov eax, 0
mov ecx, [eax]

mov eax, [esp]
mov fs:[0], eax
add esp, 8
```

这里 `mov ecx, [eax]` 会访问空指针，因为 `eax = 0`。

如果你只按普通线性思路看，会觉得：

```text
程序在这里非法访问，应该崩溃
```

但结合前面的 SEH 注册，你应该马上换个思路：

```text
不，这大概率是作者故意触发异常，好把控制权送进 seh_handler
```

此时最合理的动作不是反复盯着崩溃点，而是记下 `seh_handler`，直接在它下面断，让异常继续，然后看异常处理器里到底做了什么。很多真正逻辑、解密动作、环境修复，都会藏在那里。

---

## 十一、SEH 为什么特别适合做控制流隐藏？

因为它天然就带着三层伪装。第一层是它看起来像“错误”，不像“跳转”，这对静态分析很不友好；第二层是它借的是系统机制，不是作者自己明晃晃写一条 `jmp`，而是借操作系统分发异常偷偷转移；第三层是它会打乱调试节奏，异常一来，弹窗、暂停、处理选项一堆，人的思路也容易被带乱。所以 SEH 混淆不一定逻辑多深，但它真的很擅长把人绕晕。

---

## 十二、初学者最常见的几个坑

### 1. 看到异常就默认程序结束

有 SEH 的场景里，这个判断经常是错的。

### 2. 只看普通跳转，不看异常路径

真正逻辑可能根本不在 `jmp` 后面，而在异常处理函数里。

### 3. 没注意 `FS:[0]`

这几乎是 32 位 SEH 分析的重要路标。

### 4. 调试器一弹异常就直接停止分析

很多时候你需要做的是“让它按设计继续一次”，看处理器接不接。

### 5. 只盯触发异常的指令，不盯处理函数

触发点常常只是烟雾弹，真正要看的地方是 handler。

---

## 十三、把 SEH 控制流想成“暗门”

如果普通控制流是走走廊，那 SEH 控制流就像墙上的暗门。

你看着主路以为已经走到尽头了，实际上作者在旁边留了个机关：

```text
触发异常 -> 墙开了 -> 从暗门进真正房间
```

而这个机关的触发方式，往往故意设计得像“事故现场”：

- 除零
- 访问非法地址
- 非法指令

所以分析 SEH 最重要的意识转变就是：

**看到异常，不要只想着“怎么崩了”，还要问“它是不是故意想从这里拐弯”。**

---

## 十四、结语

SEH 本来是错误处理机制，但在逆向对抗里，它完全可以变成控制流伪装工具。作者先通过 `FS:[0]` 注册异常处理器，再故意制造一次异常，让系统替他把执行流送进隐藏逻辑。

这套玩法的精髓在于：它不和你正面打招呼。它不写明白的 `jmp true_logic`，而是装作“哎呀我崩了”，等你放松警惕后再偷偷把路拐过去。

所以碰到这类样本时，最关键的不是背多少异常代码，而是建立正确观察方式：

- 留意 `FS:[0]` 和 handler 注册
- 对故意制造的异常保持警觉
- 在异常处理函数下断，而不是只盯崩溃点

当你开始习惯从“异常之后”去看控制流时，SEH 这条暗道就没那么神秘了。

它只是没走寻常路而已。
