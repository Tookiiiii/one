# 第13章 int 指令

## 本章主线

第 12 章讲的是“CPU 内部触发的中断”，这一章讲的是“程序自己主动触发中断”。这就是 `int` 指令。

## 按小节过一遍

### 13.1 `int` 指令

- `int n` 的含义是：主动引发一个 `n` 号中断过程
- 它会像真实中断一样，借助中断向量表跳到对应处理程序
- 所以 `int` 本质上是一种软件调用门

### 13.2 供应用程序调用的中断例程

- 书里开始强调：中断不只是一种硬件异常机制，也可以被设计成“供别人调用的服务接口”
- 这其实就是后面 BIOS、DOS 中断服务的思想

### 13.3 深入理解 `int/iret/栈`

- `int` 为什么能返回？
- 因为它会在进入时保存返回环境，`iret` 再从栈恢复
- 所以这里和 `call/ret` 一样，也必须用“栈视角”理解

### 13.4 ~ 13.7 BIOS、DOS 中断例程及应用

- BIOS：更偏底层、偏硬件
- DOS：更偏操作系统服务
- 它们都通过中断号 + 功能号的方式对外提供服务
- 这就是经典 DOS 编程的大门

## 本章最该记住的结论

- `int` 是软件主动发起的中断过程。
- BIOS 和 DOS 中断，本质上是系统提供的一组服务接口。
- 只要明白“中断号 + 功能号 + 参数位置 + 返回约定”，你就能调用现成系统服务。

## 最小例子

```asm
mov ah, 4ch
mov al, 0
int 21h
```

这里的意思就是调用 DOS 的 `21h` 中断服务中的 `4Ch` 功能，正常结束程序。

## 易错点

- 易错 1：把 `int` 只理解成“错误/异常”
- 它也可以是主动调用系统服务
- 易错 2：只记中断号，不记功能号和参数放哪
- 易错 3：调用服务后不看返回结果和相关寄存器变化

## 线下赛/逆向视角

- 老式程序里看到 `int 21h/int 10h/int 16h/int 13h`，就和看到 API 调用差不多
- 只不过它们不是现代导入表 API，而是中断接口
