# 加密算法的“照妖镜”：魔改 Base64 与常见密码算法特征识别

逆向里有一种非常让人上头的时刻：你还没把整段逻辑完全看懂，只是扫了几眼汇编，突然心里冒出一句话:

“等等，这玩意儿怎么这么像 RC4？”

如果这句判断是对的，你能省下来的时间可能不是十分钟，而是大半天。因为逆向并不总是比谁更能硬啃汇编，有时候真正拉开效率差距的，是**你能不能快速识别出代码背后那套熟悉的算法轮廓**。很多时候并不是非得把整段都嚼碎，先认出它在干啥，事情就已经好办一大半了。

这一篇就来聊聊这个“照妖镜”能力：不讲太深的数学证明，重点讲**在汇编、伪代码、数据形态里，怎么认出常见算法**。我们先从最容易遇到的 Base64 开始，再看 RC4 和 TEA，最后讲讲“魔改”之后怎么处理。

---

## 一、为什么要训练“算法识别”能力？

因为你在逆向里经常会碰到这种情况：程序里摆着一段看不懂的处理函数，输入输出又很像编码或者加密。如果这时候你完全不知道它是什么，那通常只能从头跟寄存器、看循环、猜变量、慢慢推；但如果你一眼觉得“这玩意儿怎么这么像 RC4”，那你立刻就会去验证有没有 256 长度的状态数组、有没有 KSA 和 PRGA 那两套味道、有没有交换 `S[i]`、`S[j]`、有没有 `S[(S[i] + S[j]) & 0xFF]`。这两种做题速度差得还是挺明显的。

所以这篇文章的核心目标不是“教你设计密码算法”，而是“教你一眼看出它大概是谁”。研究算法当然重要，但如果一个函数名、一个循环结构、一个魔数已经把答案快贴你脸上了，那就没必要和它硬耗。

---

## 二、先从最温和的 Base64 讲起

严格说，Base64 不是加密，而是编码。但在逆向里，它出现频率高得离谱，所以必须会认。因为真碰到题的时候，你未必需要把它从位运算层面手算一遍，但你得尽快判断“哦，这就是它”。

Base64 干的事情其实很简单：每 3 个字节，也就是 24 位原始数据，切成 4 组，每组 6 位，然后拿这 6 位去查一张 64 字符的表。

标准表长这样：

```text
ABCDEFGHIJKLMNOPQRSTUVWXYZabcdefghijklmnopqrstuvwxyz0123456789+/
```

如果输入字节数不是 3 的倍数，就会在结果末尾用 `=` 补齐。

这就是为什么 Base64 很有辨识度：

- 它天然有查表行为
- 输出字符集很窄
- 常见尾部会出现 `=` 或 `==`

---

## 三、最简单的 Base64 例子

比如字符串 `Man`：

```text
M = 0x4D = 01001101
a = 0x61 = 01100001
n = 0x6E = 01101110
```

拼起来是 24 位：

```text
010011 010110 000101 101110
```

转换成十进制索引：

```text
19 22 5 46
```

再去标准 Base64 表里查：

```text
19 -> T
22 -> W
5  -> F
46 -> u
```

于是：

```text
"Man" -> "TWFu"
```

这个例子非常重要，因为它把 Base64 的两个本质动作直接摆你脸上了：前面是位运算拆组，后面是码表查表。所以你在逆向里如果看到某段逻辑大量出现 `shr`、`shl`、`and 0x3F`，还对一张固定 64 长度表做索引，那就该想到 Base64 了，别再硬往别的方向想。

---

## 四、在汇编里怎么认 Base64？

Base64 的识别点比很多人想象中更明显。最扎眼的当然是一张 64 字符表。静态分析时，如果你在只读数据区看到：

```text
ABCDEFGHIJKLMNOPQRSTUVWXYZabcdefghijklmnopqrstuvwxyz0123456789+/
```

这几乎就是在举牌子自报家门。哪怕不是标准表，只要你看到长度接近 64、字符分布又集中在字母数字和少量符号上，也很可疑。再往下看，如果处理节奏明显是 3 字节转 4 字节，伪代码通常会像这样：

```c
out[0] = table[in[0] >> 2];
out[1] = table[((in[0] & 0x03) << 4) | (in[1] >> 4)];
out[2] = table[((in[1] & 0x0F) << 2) | (in[2] >> 6)];
out[3] = table[in[2] & 0x3F];
```

这里最该记的就是 `>> 2`、`<< 4`、`<< 2` 和 `& 0x3F` 这些动作，它们就是 Base64 非常典型的味道。再进一步，如果输出尾部还有 `=` 补齐逻辑，比如：

```c
if (remain == 1) out[2] = '=', out[3] = '=';
if (remain == 2) out[3] = '=';
```

那就更稳了。

---

## 五、Base64 最烦人的变种：魔改码表

现实里最常见的“魔改算法”，其实不是改得多高深，而是把标准 Base64 码表换了。看着吓人，实际往往只是给你加了层节目效果。

例如标准表：

```text
ABCDEFGHIJKLMNOPQRSTUVWXYZabcdefghijklmnopqrstuvwxyz0123456789+/
```

被改成：

```text
QWERTYUIOPASDFGHJKLZXCVBNMabcdefghijklmnopqrstuvwxyz0123456789+/
```

这样一来，处理流程还是 Base64，位运算结构还是一样，只是最后查表映射变了。如果你只盯结果字符串，确实会觉得“这怎么解不出来”；但如果你认出了“流程仍然是 Base64”，问题就一下简单很多：**把那张表抠出来，按它的顺序逆映射回去。**

---

## 六、魔改 Base64 怎么还原？

思路非常直接：

1. 找到程序里那张自定义码表
2. 建立“自定义字符 -> 索引值”的逆向映射
3. 按 Base64 规则还原 6 位分组
4. 再拼回原字节流

一个最小脚本思路大概这样：

```python
custom = "QWERTYUIOPASDFGHJKLZXCVBNMabcdefghijklmnopqrstuvwxyz0123456789+/"
rev = {ch: i for i, ch in enumerate(custom)}
```

后面再按 Base64 的 4 个字符一组，取索引、拼位、还原原始字节即可。

这里真正重要的不是脚本长什么样，而是你的判断路径：

**先认出它还是 Base64，再处理它“表被换了”这个小动作。**

---

## 七、再看一个逆向高频选手：RC4

RC4 之所以好认，不是因为它数学简单，而是因为它代码结构太有个人风格了。

RC4 的核心其实就两部分，KSA 负责密钥调度，PRGA 负责生成伪随机流。它还有一个非常标志性的状态数组：

```text
S[256]
```

然后它会反复做 `i = (i + 1) & 0xFF`、`j = (j + S[i]) & 0xFF`、交换 `S[i]` 和 `S[j]`，再取 `S[(S[i] + S[j]) & 0xFF]` 作为密钥流字节去和数据异或。你看，RC4 的特征其实已经在自我介绍了：**256、交换、双索引、异或。**

---

## 八、最简单的 RC4 例子

别上来就管它几百行。先理解最简单模型：

### 初始化

```c
for (i = 0; i < 256; i++) {
    S[i] = i;
}
```

这一步很关键，因为你在逆向里如果看到一个 256 次循环，把数组填成 `0,1,2,...255`，就该非常警觉。

```c
j = 0;
for (i = 0; i < 256; i++) {
    j = (j + S[i] + key[i % keylen]) & 0xFF;
    swap(S[i], S[j]);
}
```
这部分就是 KSA。后面再接上：

```c
i = j = 0;
while (...) {
    i = (i + 1) & 0xFF;
    j = (j + S[i]) & 0xFF;
    swap(S[i], S[j]);
    k = S[(S[i] + S[j]) & 0xFF];
    out = in ^ k;
}
```

别背公式，记味道就行。

---

## 九、在汇编里怎么认 RC4？

认 RC4 的时候，不必拆得太细。你先看有没有特别扎眼的 256 次循环，再看是不是有个 256 大小的数组被频繁交换。很多 RC4 伪代码一眼望去就是不停地：

```c
tmp = S[i];
S[i] = S[j];
S[j] = tmp;
```

再往后，如果输出阶段明显是“生成一个字节，再异或一个字节”，比如：

```c
buf[n] ^= k;
```

或者：

```c
out[n] = in[n] ^ k;
```

那就更像了。再补一个细节，很多索引都会带 `& 0xFF`，因为 RC4 的状态空间就在 0 到 255 之间打转。所以只要你看到 256 长度数组、两个游标变量 i/j、反复交换、最后又和数据异或，这玩意基本就八九不离十了。

---

## 十、TEA 系列为什么也常被一眼认出来？

因为它有一个逆向圈里几乎成了门牌号的常数：

```text
0x9E3779B9
```

这就是 TEA 里常见的 Delta 常数。TEA、XTEA、XXTEA 虽然细节不同，但通常都会留下几样很显眼的东西：成对的 32 位整数操作、多轮循环、加减异或移位混在一起，还有 `0x9E3779B9` 这种门牌号一样的常数。它不像 Base64 那样有码表，也不像 RC4 那样有个 256 大数组，但它会在“常数 + 轮函数结构”上露馅。

---

## 十一、最简单的 TEA 轮函数长什么样

看一个典型轮函数味道：

```c
sum += 0x9E3779B9;
v0 += ((v1 << 4) + k0) ^ (v1 + sum) ^ ((v1 >> 5) + k1);
v1 += ((v0 << 4) + k2) ^ (v0 + sum) ^ ((v0 >> 5) + k3);
```

它很有辨识度，左移 4、右移 5、加法和异或交织在一起，再配一个每轮递增的常数。你在反汇编里如果看到一个循环不断做 `shl reg, 4`、`shr reg, 5`、`add`、`xor`，旁边还老跟着某个固定常数转，那 TEA 家族就很值得怀疑了。

---

## 十二、TEA 被“魔改”了怎么办？

很多题目不会傻乎乎把标准 TEA 原样摆着，常见的改法无非是把 Delta 常数改了、轮数改了、密钥索引顺序调一调，或者把加减方向变一下。但它一般不会把整个结构改得面目全非，因为那样作者自己也容易写崩。所以识别时别过度死盯 `0x9E3779B9` 这个数字本身，更该看整体结构：是不是两个 32 位分组交替更新，是不是有多轮累加常数，是不是混着 `<<4` 和 `>>5`，是不是每轮都依赖 sum 和 key。如果这些骨架都还在，只是常数不同，那大概率就是“换皮 TEA”。

---

## 十三、一个很实用的识别顺序

拿到一段可疑编码/加密函数时，我通常会先这么看：

### 1. 先看数据形态

- 输出是不是可打印字符？
- 有没有明显 `=`？
- 长度有没有 4 对齐、8 对齐、16 对齐特征？

### 2. 再看是否存在静态表

- 64 长度字符表：优先怀疑 Base64
- 256 大小状态表：优先怀疑 RC4
- 没表但有固定轮函数：考虑 TEA/AES 一类

### 3. 再看循环结构

- 3 字节转 4 字节：像 Base64
- 256 初始化 + 交换：像 RC4
- 32 轮左右 + 常数累加：像 TEA

### 4. 最后看关键常数和关键操作

- `& 0x3F`：Base64 味很重
- `& 0xFF` + swap：RC4 味很重
- `0x9E3779B9`、`<<4`、`>>5`：TEA 味很重

这个顺序比一上来硬啃细节更稳。

---

## 十四、为什么“识别”比“还原”更重要？

因为很多时候，真正难的不是写个解密脚本，而是确定你该写哪一种脚本。

比如一旦认出是：

- Base64：你多半只需要抠码表
- RC4：你需要找 key 和流程位置
- TEA：你需要抠轮函数、轮数、Delta、key

路径完全不同。

所以所谓“照妖镜”能力，其实不是让你一眼写出答案，而是让你一眼排除大量错误方向。说得再直白一点，就是别在不该深挖的地方死磕，把时间花在真正卡人的部分上。

---

## 十五、初学者常见误区

### 1. 看到“加密函数”就默认是 AES

很多题目根本没上这么重的算法，反而是 Base64 魔改、异或、RC4、TEA 这些更常见。

### 2. 只盯输出，不看处理结构

输出长得奇怪，不代表算法高深；很可能只是表被换了。

### 3. 过度依赖常数，不看整体流程

比如 TEA 常数被改了，你如果只会认 `0x9E3779B9`，就容易错过。

### 4. 不区分编码和加密

Base64 更多是编码，不负责保密。别一看到乱码就觉得“这是高级密码学”。

### 5. 不敢写小脚本验证

逆向里很多判断，最好的确认方式就是拿样本数据自己试一次。

---

## 十六、结语

算法识别这件事，说到底是一种“结构直觉”。

你不是在背题库，而是在训练自己看到下面这些信号时立刻联想：

- 64 字符查表和 `=`：像 Base64
- 256 状态数组和交换：像 RC4
- 轮函数、位移、常数：像 TEA

一旦这种直觉养成，很多原本看起来“黑乎乎一团”的代码，会突然开始有轮廓。你不再是漫无目的地逐条翻译汇编，而是在和熟悉的模式对接。

逆向里最省时间的能力，往往不是手速，而是识别力。

而这面“照妖镜”，越早练出来越值钱。
