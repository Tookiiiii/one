# 逆向里的“躲猫猫”：自修改代码（SMC）原理与动静结合破解

有些程序第一次扔进 IDA 里，画面会非常劝退：一段看起来像代码的地方，全是 `00 00 00`，或者夹着一堆完全不顺眼的乱码。你盯着它，它也盯着你，场面一度十分尴尬，我去，hyw啊

这时候先别急着气笑了。程序未必坏了，也未必是 IDA 抽风了。更常见的情况是：你看到的压根不是“最终要执行的代码”，而是**还没来得及被程序自己改写出来的半成品**。这就是自修改代码，Self-Modifying Code，简称 SMC。

来来来, 我们先从最朴素的原理讲起，再看它在逆向里为什么烦人，最后再落到实际分析里：怎么静态找解密逻辑，怎么动态卡住它“现原形”的那一刻。思路上也尽量实用一点，不和乱码硬较劲，先抓真正卡住我们的部分。

---

## 一、什么叫自修改代码？

字面意思其实非常直白：**程序在运行过程中，去改自己后面要执行的那段代码**。

正常程序的执行模型大概是这样：

```text
磁盘上的代码 -> 装载进内存 -> CPU 按顺序执行
```

而 SMC 程序多了一步：

```text
磁盘上的“伪装代码” -> 装载进内存 -> 运行解密/改写逻辑 -> 得到真正代码 -> CPU 执行真正代码
```

也就是说，你在静态分析时看到的那段 `.text`，有时候只是“伪装层”。真正的逻辑要等程序跑起来以后，自己把自己修一遍，才会露出来。

从攻击者或者加壳作者的角度看，这招很划算：

- 静态分析器看到的是假内容，阅读成本立刻暴涨。
- 签名匹配变难，因为关键代码可能根本不直接落在文件里。
- 动态分析时如果你下断点太晚，程序已经执行过去了；下断点太早，又可能只踩到一堆解密循环。

所以 SMC 本质上不是“代码会变魔术”，而是**把“生成真实代码”的过程推迟到运行时**。

那么与之对应的应对手段也便呼之欲出了

---

## 二、最简单的 SMC 例子：先加密，再现场解密

先看一个极简思路，不追求能直接编译，只追求把原理说透：

```c
unsigned char encrypted_code[] = {
    0x33 ^ 0xAA, 0xC0 ^ 0xAA, 0x40 ^ 0xAA, 0xC3 ^ 0xAA
};

int main() {
    for (int i = 0; i < 4; i++) {
        encrypted_code[i] ^= 0xAA;
    }

    ((void(*)())encrypted_code)();
    return 0;
}
```

这里发生了什么？

1. 原本真正的机器码是：`33 C0 40 C3`
2. 它在文件里没有直接出现，而是先和 `0xAA` 异或过
3. 程序运行后，先循环一遍，把每个字节再异或一次
4. 异或两次回到原值，于是真正代码被还原出来
5. 最后把这块内存当函数执行

如果你静态看文件，看到的是“加密后的字节”；如果你动态跟到解密后，看到的才是“能正常反汇编的代码”。

这个例子虽然简陋，但它已经具备了 SMC 的核心结构：

- 有一块“被保护的代码”
- 有一个“解密/改写例程”
- 有一个“切换到真实逻辑”的时机

后面再怎么花哨，肯定都绕不开这三样

## 三、SMC 到底改的是哪里？

当我第一次听到“程序改自己的代码”,我就感觉这也太妙了吧?! 这种“程序先把真货藏起来，运行时再自己掏出来”，简直天生就是逆向的料子嘻嘻

程序通常会把一段数据放进内存，然后：

- 直接改 `.text` 节里的内容
- 把加密代码解到新申请的可执行内存中
- 先 `VirtualProtect` 改页权限，再覆盖原指令
- 或者一边解密一边跳过去执行

关键点只有两个：

## 1. 它手里得有“原材料”

这个原材料可能是：

- 被异或/加密后的字节数组
- 压缩数据
- 多段碎片化的机器码
- 甚至是通过算法临时算出来的指令流

## 2. 它得有“施工权限”

代码页默认未必可写，所以常见套路是：

```c
VirtualProtect(target, size, PAGE_EXECUTE_READWRITE, &oldProtect);
memcpy(target, decrypted_buf, size);
```

看到 `VirtualProtect`、`WriteProcessMemory`、`memcpy`、循环异或、按字节改内存这些组合时，就该警觉一点：这程序多半没打算老老实实把真实逻辑直接给你看。

## 四、静态分析时，怎样闻到 SMC 的味道？

SMC 最烦的地方，就是你很可能还没看到真代码。但“没看到”不等于“没线索”。

常见信号大概有这些：

### 1. 某段区域明明会被跳转过去，但当前反汇编结果很怪

比如：

```asm
call decrypt_stub
jmp 401000
```

结果 `401000` 这个位置现在看过去全是：

```asm
db 00h, 00h, 00h, 00h ...
```

这就很像“目标代码还没生成”。

### 2. 存在明显的字节级循环处理

例如一类很典型的伪代码：

```c
for (i = 0; i < len; i++) {
    buf[i] ^= key;
}
```

或者：

```c
for (i = 0; i < len; i++) {
    buf[i] = rol(buf[i], 3) + 7;
}
```

这种循环如果处理的目标地址正好靠近代码段，就要重点关注。

### 3. 出现改内存权限的 API

特别是：

- `VirtualProtect`
- `VirtualAlloc`
- `NtProtectVirtualMemory`
- `VirtualProtectEx`

这些可是大明星 !
如果某段逻辑先申请/改权限，再做大段内存写入，最后跳转过去，基本已经把"我要自修改了"写在脸上了

### 4. 某些关键函数结尾不是正常返回，而是突然跳去一块先前不可读的区域

这很像“舞台搭好了，演员终于准备登场”,这时候我们再神兵天降!一切谜题就已全部落入手中了!

## 五、静态阶段该怎么做：先找“解密函数”

既然暂时看不到真实代码，那静态分析的目标就不应该是硬啃乱码，而应该改成：

**先把负责还原代码的那段逻辑找出来。**

这一步其实特别像做题时找“关键块”。哪一段真正决定它会不会现原形，就先盯哪一段，别急着把每个字节都研究个遍。

一个比较实用的思路是：

### 第一步：找可疑 API 和可疑循环

在 IDA 里先看：

- 导入表中有没有 `VirtualProtect`
- 有没有大段 `memcpy`、`rep movs`
- 有没有按字节/按 dword 处理缓冲区的循环

### 第二步：看处理目标是不是代码区

如果某个循环处理的地址范围正好落在：

- 原 `.text` 节
- 即将跳转的地址附近
- 新申请的可执行内存

这可是好事啊 !
那它极可能就是解密器 !

### 第三步：推密钥和算法

有些简单 SMC，静态阶段就能直接把密钥看出来。比如：

```asm
xor byte ptr [esi+ecx], 55h
```

这就是作者手下留情了。我们甚至可以直接写个脚本离线还原

更复杂一点的情况，密钥可能：

- 来自时间
- 来自环境校验
- 来自前面算出来的寄存器值
- 分块变化，不是固定单字节

这时静态分析的目的就从“直接解密”变成“弄懂解密函数输入输出关系”，为动态调试做准备。

## 六、动态阶段为什么好用：因为真代码迟早要落地

SMC 再会藏，它也得把真实代码摆到内存里，否则 CPU 没法执行。

这就是动态分析最大的底气：**你不一定能提前知道答案，但你总能在它真正执行之前抓住它。**

最典型的问题是：到底该在什么时候截住？

如果你在程序入口就一路 F7，通常会非常痛苦，因为：

- 解密循环可能跑很久
- 你容易在大量无关指令里迷路
- 代码刚解完就被立刻跳走，错过最佳观察窗口

所以更好的办法是：**盯“被写入的目标地址”而不是盯“当前执行流”**。

---

## 七、硬件断点为什么特别适合抓 SMC？

处理 SMC 时，硬件断点非常实用，因为它可以对某个地址的“读/写/执行”做精准监控，而且不像软件断点那样需要直接改目标代码。

对 SMC 来说，最常见的用法是：

- 先定位真实代码将要落在哪一片内存
- 对那片区域设置“写入断点”
- 让程序继续运行
- 一旦解密器开始往这里写字节，调试器立刻断下

这样你就能看到, 谁在写? 写了多少? 写完后跳到哪里?

如果目标区域已经大体清楚，还可以在最终执行位置下“执行断点”，用来卡住 OEP(Original Entry Point)

这里的核心思想很重要：

**不是傻乎乎跟完整个解密过程，而是用断点把“真代码出现”和“开始执行”的关键瞬间钉住。**

## 八、一个很常见的实战节奏

用 IDA + x64dbg 分析一段 SMC，通常可以按这个顺序来。顺着这个节奏走，思路会比一上来 F7 到天荒地老要爽得多：

### 1. 在 IDA 里先认框架

你重点看三件事：

- 解密函数在哪里
- 目标地址在哪里
- 解密完成后跳去哪

这一步不求把每行都看懂，只求把“谁改谁”关系理出来。

### 2. 在 x64dbg 里运行到解密前

可以先在：

- `VirtualProtect`
- 可疑循环入口
- 返回后紧接着的跳转点

这些地方下断。

### 3. 给目标地址下硬件写入断点

比如你发现最终要跳到 `00401000`，那就盯住这块区域。

### 4. 放程序跑

一旦断下，就检查内存窗口里对应位置的字节是不是已经从乱码变成了正常指令。

### 5. 确认 OEP

如果解密后出现了很像正常函数序言的代码，比如：

```asm
push ebp
mov ebp, esp
sub esp, xx
```

或者开始出现正常 API 调用、字符串引用、业务判断，那通常说明真逻辑已经到了

### 6. Dump 内存重新分析

既然现在内存里已经有“真实代码版本”，就没必要继续盯着假壳看。把这时的代码 Dump 下来，再扔回 IDA 分析，效率会高很多。

---

## 九、为什么 Dump 之后还要“重新分析”？

因为你在最开始看到的是“施工前现场”，Dump 下来的是“施工完成现场”,两者的可读性完全不是一个量级

重新分析的好处主要有：

- 函数边界更正常
- 交叉引用更完整
- 字符串和 API 关系更清晰
- 很多原本反汇编错误的位置会恢复正常

当然，Dump完可还没完,和脱壳有点像,我们通常还得：

- 修重定位
- 修导入
- 处理后续二次解密
- 重新命名函数和结构体

这样我们终于可以回到正常的简单程序逆向了!!

## 十、一个最小分析例子：异或解密后跳转

假设你在反汇编里看到这样一段伪汇编：

```asm
lea esi, encrypted_block
mov edi, 401000h
mov ecx, 100h

decode_loop:
mov al, [esi]
xor al, 5Ah
mov [edi], al
inc esi
inc edi
loop decode_loop

jmp 401000h
```

这个例子已经把 SMC 思路写得很直白了：

- `encrypted_block` 是加密数据源
- `401000h` 是被还原出来的目标代码区
- 解密方式是 `xor 5A`
- 循环结束后立刻 `jmp 401000h`

如果你是静态分析：

- 先记录 `401000h`
- 确认 `xor 5A` 是不是唯一处理逻辑
- 离线写脚本尝试还原

如果你是动态分析：

- 在 `401000h` 附近设硬件写入断点
- 跑到断下时看内存
- 解密完后在 `401000h` 执行断下

这是最小可用模型。真实样本可能把这套流程拆得很碎，但本质差别并不大。

## 十一、初学者最容易踩的几个坑

### 1. 把“乱码”直接当损坏文件

很多时候不是文件坏了，而是你看得太早了。

### 2. 只盯当前 EIP，不盯目标内存

SMC 的关键不是“现在执行到哪”，而是“哪块内存正在变成真正代码”。

### 3. 过度迷信一次性静态还原

有的样本密钥依赖运行环境，纯静态不一定划算。看不动就切动态，不丢人，反而专业。

### 4. 抓到 OEP 之后不 Dump

很多人辛苦半天终于看见真代码，然后继续在壳层里硬啃。其实这时候最该做的是把成果固化下来。

### 5. 忽略二次自修改

有些程序不是解一次就完，而是：

```text
第一层解密 -> 执行一段 -> 再解第二层 -> 再跳转
```

如果发现刚看懂一点，代码又突然“变脸”，那就说明它可能是多阶段 SMC
