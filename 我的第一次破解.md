# Princess Synergy 整合包卡密验证分析

## Summary

目标并不是原游戏自带的 DRM，而是整合包里混入的第三方验证层。最初从 `Tool/MTool` 外壳入手，随后因为“直接启动 `PS.exe` 仍然弹出卡密验证”这一现象继续回溯，最终确认当前真正拦截启动流程的逻辑已经被直接前插进根插件 `www/js/plugins/JsScript190Set.js`，其前 30 行是一段独立的 `Key Vault` 联网校验代码，后半段才是正常的 Live2D 插件正文。

本次分析主要使用了 Windows PowerShell、Python 3.13、`pefile` 和字节串检索，不涉及补丁、绕过或修改样本。

## Solution

### Step 1: 先还原整合包最外层的启动链

第一轮静态分析先从分发者提供的启动脚本入手，结果很快确认这不是“只启动游戏”的纯净环境，而是一条明显的第三方装配链。

- `与工具一同启动.bat` 第 18 行会把 `Tool\loaders\mzHook32.dll` 注入到 `PS.exe`
- 同一脚本第 19 行会继续启动 `Tool\MTool.exe`
- `Tool/package.json` 中把这套壳标识为 `MToolClient`
- `Tool/serverURI` 指向本地地址 `127.0.0.1:11799`
- `Tool/www/rawres/LangTL.json` 里可以直接命中 `applySupportsKey`、`pleaseInputSupportsKey`、`supportExpired`、`cantUseThisKey`
- `Tool/www/loader.bin.*` 中能提取出 `HelloMtool`、`hiMtool`、`trsdev.mtool.app`、`cntrs.mtool.app:1300`

这一步能说明两件事：

1. 整合包确实包含了一套完整的 `MTool` 生态。
2. “卡密/支持者 key” 语义首先出现在第三方工具链里，而不是原游戏剧情资源里。

顺便验证 `MTool.exe` 的身份时，还发现它本质上是改名后的 NW.js 运行时。它和同目录 `nw.exe` 的 SHA-256 完全相同：

```text
d48ae12a7b8bea18ec76770f08c2d0098b4d036bc2ba905526c1705e291f6827
```

所以 `Tool/MTool.exe` 不是传统原生启动器，而是一个基于 NW.js 的二次包装 UI 壳。

### Step 2: 用“单开 PS.exe 仍弹卡密”作为关键 pivot

如果验证只存在于 `Tool` 外壳，那么单独双击 `PS.exe` 理论上应该绕开它。但实际观察是：即使不走批处理脚本，`PS.exe` 仍然会进入卡密页面。

这说明上一轮结论只能解释“外层工具链存在”，还不能解释“当前实际拦截点在哪里”。于是第二轮分析把重点从 `Tool` 目录移回根游戏资源：

- `PS.exe` 是标准 NW.js 启动器，会直接加载根 `www/index.html`
- 根 `www/index.html` 本身就被改过，尾部额外插了一段会在根目录创建论坛标记文件的脚本
- 插件清单 `www/js/plugins.js` 中，`JsScript190Set` 处于启用状态

真正的突破点在 `www/js/plugins/JsScript190Set.js`：

- 第 1 到 30 行是一段完整的自执行校验器
- 第 31 行开始才出现正常的 RPG Maker / Live2D 插件头注释
- 也就是说，这个文件被人“借壳混入”了前置验证逻辑

这一点直接解释了为什么单开 `PS.exe` 仍然会弹卡密：因为 NW.js 启动后会照常加载根插件，而根插件里已经被塞进了验证层，不需要 `Tool\MTool.exe` 另起窗口也能完成拦截。

### Step 3: 复原根插件中的 Key Vault 验证流程

把 `JsScript190Set.js` 前 30 行拆开后，验证过程基本可以完整复原。

首先是常量定义：

- `_a='https://keqing.us.ci/api/verify'`，联网校验地址
- `_b='_runtime.dat'`，本地缓存文件名
- `_d='__kv_rt'`，浏览器本地存储键名
- `_e=30`，默认许可有效期 30 天

接着是本地状态流：

- `_sv()` 把许可状态编码后写入 `_runtime.dat` 或 `localStorage`
- `_ld()` 读取并解码缓存
- `_vl()` 验证缓存是否存在、是否过期、时间戳是否合理
- 如果 `_vl()` 返回真，脚本直接 `return`，不会继续构造遮罩

然后是联网校验层 `_send()`。它为了尽可能“打通网络”，同时准备了四条传输路径：

1. Node 的 `https.request`
2. `curl.exe`
3. `powershell.exe` + `Invoke-WebRequest`
4. 浏览器侧 `XMLHttpRequest`

只要其中任意一条拿到合法 JSON 响应，流程就会继续。失败时会自动重试，最长 60 秒。超时后的提示文案也写死在脚本里：

```text
你的网络环境无法校验，请私信 up 主来帮你解决
```

最后是界面拦截：

- `_build()` 在页面最上层插入全屏遮罩 `kv-block-overlay`
- 标题就是“卡密验证”
- 输入框占位符为 `KV-XXXXX-XXXXX-XXXXX-XXXXX`
- 成功后会把到期时间、key 哈希、当前时间戳写回缓存
- 失败则显示 `invalid_key`、`key_used`、`key_expired`、`key_revoked` 等错误文案

因此，当前样本中的“卡密验证”并不是单纯的 DLL hook，也不是只存在于 `Tool` 外壳中的附属窗口，而是已经被直接前插到根游戏插件的联网校验层。

## Reproduction Script

下面这段脚本只做证据提取，不修改样本。它会同时验证外层 `MTool` 链和根插件中的 `Key Vault` 注入链。

```python
from pathlib import Path
from hashlib import sha256
import json

game = Path(r"C:\Users\lTOOKI\Desktop\codex\Princess Synergy 汉化整合版")
tool = game / "Tool"
root_plugins = game / "www" / "js"
target_plugin = root_plugins / "plugins" / "JsScript190Set.js"


def sha256_file(path: Path) -> str:
    return sha256(path.read_bytes()).hexdigest()


def hit_lines(path: Path, needles):
    text = path.read_text(encoding="utf-8", errors="ignore")
    out = []
    for idx, line in enumerate(text.splitlines(), 1):
        if any(n in line for n in needles):
            out.append((idx, line.strip()))
    return out


print("[1] 外层启动链")
launcher = game / "与工具一同启动.bat"
for idx, line in hit_lines(launcher, ["inject.exe", "mzHook32.dll", "MTool.exe"]):
    print(f"{idx}: {line}")

print("\n[2] MTool 运行时身份")
mtool = tool / "MTool.exe"
nw = tool / "nw.exe"
print("MTool.exe SHA256:", sha256_file(mtool))
print("nw.exe    SHA256:", sha256_file(nw))
print("same hash:", sha256_file(mtool) == sha256_file(nw))

print("\n[3] Tool 侧 supporter / key 痕迹")
lang = tool / "www" / "rawres" / "LangTL.json"
for idx, line in hit_lines(
    lang,
    [
        "applySupportsKey",
        "pleaseInputSupportsKey",
        "supportExpired",
        "cantUseThisKey",
    ],
):
    print(f"{idx}: {line}")

print("\n[4] 根插件启用状态")
plugins_js = root_plugins / "plugins.js"
for idx, line in hit_lines(plugins_js, ["JsScript190Set"]):
    print(f"{idx}: {line}")

print("\n[5] 根插件前置验证层")
head = target_plugin.read_text(encoding="utf-8", errors="ignore").splitlines()[:30]
for idx, line in enumerate(head, 1):
    if any(
        marker in line
        for marker in [
            "keqing.us.ci",
            "_runtime.dat",
            "__kv_rt",
            "curl.exe",
            "Invoke-WebRequest",
            "卡密验证",
            "Key Vault",
        ]
    ):
        print(f"{idx}: {line.strip()}")

print("\n[6] 结论")
print("当前卡密页面的直接来源是根插件 JsScript190Set.js 前插的 Key Vault 联网校验层。")
```
